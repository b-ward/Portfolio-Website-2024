## Critical Issues

### CR-01: `setError` called inside `useMemo` â€” React rule violation

**File:** `src/components/Projects/PaceCalculator/paceCalculator.jsx:53`
**Issue:** `setError("")` is called directly inside the `calc` `useMemo` callback. State setters must never be called during render (which is what `useMemo` runs during). This triggers the "Cannot update a component (`PaceCalculator`) while rendering a different component" warning in React 18 and can cause infinite render loops or stale state in concurrent mode.

**Fix:**
```jsx
// Remove setError from useMemo entirely. Move error clearing to the event handlers
// and input onChange handlers. The memo should be pure:

const calc = useMemo(() => {
  // REMOVE: setError("")    <-- delete this line
  const d = Number(distance);
  if (!d || d <= 0) return { pace: "â€”", paceUnit: "", detail: "Enter values to see results." };
  if (seconds === null || seconds <= 0) return { pace: "â€”", paceUnit: "", detail: "Enter values to see results." };
  // ... rest unchanged
}, [distance, seconds, unit, pacePref]);

// Clear error when relevant inputs change:
useEffect(() => {
  setError("");
}, [distance, hh, mm, ss]);
```

### CR-02: Pervasive direct state mutation in `trioGame.jsx`

**File:** `src/components/Projects/Trio/trioGame.jsx:250â€“972` (throughout)
**Issue:** The class component mutates `this.state` directly throughout â€” for example:
- Line 237: `this.state.pile_card_to_get = this.state.deck[0]`
- Line 250â€“265 (componentDidMount): `this.state.deck = this.shuffle(...)`, `this.state.hand.push(...)`, `this.state.deck.splice(...)`, `this.state.CPU_hand.sort()`
- Line 269: `this.state.reset = 1`
- Lines 278â€“340 (pick_up_deck, pick_up_pile, discard_initialise, play_initialise, rearrange_initialise): multiple direct assignments via `this.state.x = ...`
- Lines 366â€“390 (add_trio_1â€“4): `this.state.trio_to_add_to = N`

Direct mutation bypasses React's reconciliation. React does not re-render when you mutate state directly â€” it only re-renders on `setState`. In this component the UI is driven largely by direct DOM manipulation (`document.getElementById(...).src = ...`) rather than React state, so the mutations "work" visually but can cause subtle race conditions, missed re-renders, and inconsistency between the DOM and React's virtual DOM. This is a pre-existing pattern that was not introduced by the migration.

**Fix:** Since a full functional rewrite of the 1000-line class component is out of scope for this review, the minimum safe fix is to audit every `this.state.x = ...` assignment that is *not* immediately followed by a `setState` call, and convert it to `setState`. The most dangerous ones are in `componentDidMount` (lines 250â€“270), where the deck shuffle and deal happen without triggering a re-render.

## Warnings

### WR-01: `trioGame.jsx` â€” potential infinite loop in `pile_refresh()`

**File:** `src/components/Projects/Trio/trioGame.jsx:233-244`
**Issue:** The `while (card_changes === 0)` loop in `pile_refresh()` has no upper-bound guard. If `this.state.pile_card_to_get` is not found in `this.state.original_deck` (e.g. because a card code has a typo or the deck is corrupted), the loop runs forever, hanging the browser tab. The same pattern exists in `hand_refresh()` (line 220) and `trio_refresh()` (line 394â€“408).

**Fix:**
```js
pile_refresh() {
  let go_through_deck = 0;
  const maxLen = this.state.original_deck.length;
  const pile_card_to_get = this.state.deck[0];
  let pile_card_pos = 0;
  while (go_through_deck < maxLen) {  // bounded by deck length
    if (pile_card_to_get === this.state.original_deck[go_through_deck]) {
      pile_card_pos = go_through_deck;
      break;  // exit as soon as found
    }
    go_through_deck++;
  }
  document.getElementById('pile').src = '../Trio/Images/' + this.state.card_image[pile_card_pos];
}
```
Apply the same pattern to `hand_refresh()` and `trio_refresh()`.

### WR-04: `topArtistsMap.jsx` â€” `useEffect` missing cleanup for GeoJSON fetch

**File:** `src/components/Projects/TopArtistsMap/topArtistsMap.jsx:81-88`
**Issue:** The `useEffect` that fetches GeoJSON data has no cleanup/abort. If the component unmounts before the fetch completes (e.g. user navigates away), the promise resolves and calls `setGeojsonData` on an unmounted component. In React 18 strict mode this produces a warning; in edge cases it can update stale state.

**Fix:**
```jsx
useEffect(() => {
  refreshAuthState();
  const controller = new AbortController();
  fetch(GEO_URL, { signal: controller.signal })
    .then((res) => res.json())
    .then((data) => setGeojsonData(data))
    .catch((err) => {
      if (err.name !== 'AbortError') console.error('Failed to load GeoJSON:', err);
    });
  return () => controller.abort();
}, [refreshAuthState]);
```

### WR-09: `paceCalculator.jsx` â€” `autoAdvance` creates new function instances on every render

**File:** `src/components/Projects/PaceCalculator/paceCalculator.jsx:100-108`
**Issue:** `autoAdvance` is a regular function defined inside the component body and returns a new closure on every render. It is passed directly as `onChange` to multiple `<input>` elements. Each render creates new function objects for all three inputs, preventing React from bailing out of re-renders on those elements. While not a correctness bug, it interacts with the `useMemo` CR-01 issue to worsen render churn.

**Fix:** Wrap `autoAdvance` in `useCallback`:
```jsx
const autoAdvance = useCallback((setter, maxLen, nextRef) => (e) => {
  const v = e.target.value.replace(/\D+/g, "");
  setter(v);
  if (maxLen && v.length >= maxLen && nextRef?.current) nextRef.current.focus();
}, []);
```

### WR-10: `trainGame.jsx` â€” solution rows built by pushing JSX into an array without stable keys

**File:** `src/components/Projects/TrainGame/trainGame.jsx:21-29`
**Issue:** The `solutionRows` array uses index-based keys (`key={i}`) for the solution `<div>` elements. When the same number is submitted twice, React reuses the old DOM nodes by key, which can cause stale content if the solution list length changes. More importantly, if `sols` could return strings with embedded HTML characters (e.g. `<` from less-than operations), those are rendered as text so no XSS risk exists, but the key strategy is still fragile.

**Fix:** Use the solution string itself as the key since solutions are unique within a result set:
```jsx
for (let i = 0; i < sols.length; i++) {
  solutionRows.push(<div key={sols[i]}>{sols[i]}</div>);
}
```

### WR-11: `NBA-Ladder.jsx` â€” unhandled case where `conferences` is empty and `selectedConference` is undefined

**File:** `src/components/Projects/NBA-Ladder/NBA-Ladder.jsx:72`
**Issue:** `conferences[0]` is used as a fallback: `conferences.find(...) || conferences[0]`. If `conferences` is an empty array (initial state before load completes), `conferences[0]` is `undefined`. The `selectedConference.teams.length` access on line 101 would then throw "Cannot read properties of undefined".

The guard on line 97 does check `!selectedConference`, so this is caught for the "no teams found" display, but the condition is `(!selectedConference || selectedConference.teams.length === 0)` which is evaluated only when `!loading && !error`. During the initial render, `loading` is `true`, so `selectedConference` is never accessed â€” safe. But if the API returns an empty `conferences` array with no error (e.g. off-season or API change), `loading` becomes false, `error` is null, `selectedConference` is undefined, and the guard on line 97 catches it correctly. The risk is low but the fallback `|| conferences[0]` is misleading since it returns `undefined` when the array is empty.

**Fix:** Simplify:
```js
const selectedConference = conferences.find((c) => c.alias === selectedConfAlias) ?? null;
```
And check `selectedConference` explicitly in the guard (already done on line 97).
